# Jira/Confluence 이슈·리스크 수집 요구사항

## 배경

문제해결 사전은 현재 Open API로 개발자가 직접 이슈를 등록하는 구조만 갖춤.
하지만 사내 이슈·리스크 대부분은 Jira(장애 티켓)와 Confluence(포스트모템 문서)에 이미 누적되어 있어,
수동 재입력 대신 기존 데이터를 자동 수집하는 구조가 핵심 가치임.

## 참여자 의견 요약

| 팀원 | 핵심 의견 |
|------|----------|
| 스케치(AI 에이전트 아키텍트) | 자동 수집 1순위, 본문 LLM 분류보다 Jira 메타데이터(이슈유형/우선순위/해결상태)를 1차 필터로 우선 사용 |
| 코어(백엔드 개발자) | Pull(배치 polling) 우선 구현, 중복 방지 key 필요, 메타데이터 필터는 JQL로 저비용 구현 가능 |
| 파이프(데이터 엔지니어) | 수집 전 데이터 현황 감사 선행, 원문/정규화 결과 분리 저장, 정밀도 보정은 실사용 검색·클릭 로그 활용 |
| 옵스(DevOps 엔지니어) | 스케줄러 기반 증분 동기화, 실패 재처리(dead-letter), 캠페인은 차단보다 넛지+측정+루틴화 조합 권장 |
| 픽셀(프론트엔드 개발자) | Jira 이슈 화면에 라벨 원클릭 추천 UI로 습관화, 공포 메시지보다 본인 이득 중심 캠페인 메시지 |

## 요구사항

### 1. 수집 방식

- **1단계(PoC)**: Jira REST API(JQL 검색), Confluence REST API(CQL 검색)를 주기적으로 polling하는 배치 수집기 구현
- **2단계**: Jira/Confluence Webhook으로 상태 변경 이벤트 즉시 수신 (보안팀과 인바운드 허용 협의 필요)
- 최초 1회 전체 백필(full backfill) 후, 이후에는 `updated` 시각 기준 증분 동기화

### 2. 수집 대상 판별 (메타데이터 우선)

라벨(`troubleshooting`, `incident`)에만 의존하면 라벨 누락분을 놓침. 본문을 LLM이 먼저 읽고
판단하는 방식은 비용·오류 가능성이 커서, 아래 순서로 판별 우선순위를 둠.

1. **1차 필터(구조적 메타데이터, 비용 0)**: Jira `이슈 유형`(Incident/Bug), `우선순위`(Critical/High),
   `해결상태`(Resolved/Fixed), SLA 브리치 여부 조합으로 1차 후보 추출 (JQL 한 줄로 처리 가능)
2. **2차 보완(LLM 본문 분석)**: 1차 필터로 걸러지지 않는 애매한 케이스(예: `Bug`로 등록됐지만
   본문에 장애 키워드가 있는 경우)에 한해 LLM 분류 적용, confidence score 부여
3. **풀 분리 저장**: confidence 높음 → `confirmed`, 애매함 → `staging` (검색에는 노출하되
   "AI 자동분류 — 검증 대기" 배지로 신뢰도 구분)
4. **정밀도 보정(실사용 로그 기반)**: 사람 검수 샘플링보다 우선, 직원이 사전에서 검색·클릭한
   로그를 정답 신호로 활용 — 검색 결과 클릭률이 낮은 `confirmed` 항목은 재검토 대상으로 플래깅
5. **분류기 버전 관리**: `is_incident`, `confidence`, `classifier_version`, `classified_by`
   (`metadata`/`llm`/`usage_feedback`) 컬럼으로 추적, 판별 기준 변경 시 재분류 비교 가능
6. **선행 검증**: 메타데이터 필터가 실제로 놓치는 케이스 비율을 데이터 현황 감사로 먼저 확인
   (예: `Bug`로 등록된 실제 장애 비율) 후 1차 필터 기준 확정
- Confluence: 특정 스페이스(예: `장애보고`, `포스트모템`) 하위 페이지는 메타데이터 신호가 약해
  라벨 + 페이지 템플릿 사용 여부를 1차 필터로 사용

### 3. 데이터 정규화

- 수집한 원문(Jira 이슈 본문, Confluence 페이지 본문)은 가공 전 원문 그대로 별도 보관 (재처리 가능성 확보)
- LLM으로 원문에서 사전 필수 5항목 추출: 현상, 문제, 영향도, 해결방안, 조치결과
- 5항목 중 일부가 원문에 없으면 "추출 실패" 상태로 표시하고 사람 검수 큐로 전달 (자동 등록 금지)

### 4. 라벨링

- 1차: Jira 라벨/Confluence 페이지 라벨을 그대로 가져와 매핑
- 2차: LLM이 본문 기반 보완 라벨 제안 → 사람 승인 후 확정 (자동 라벨만으로 검색 품질 보장 불가)

### 5. 중복 방지 및 매핑

- `jira_issue_key`, `confluence_page_id`를 unique key로 사용
- 동일 key 재수집 시 신규 등록이 아닌 기존 이슈 갱신으로 처리

### 6. 운영 안정성

- 동기화 작업은 스케줄러(Cron 등)로 운영, 실패 시 알림 채널로 통보
- API 장애/Rate Limit 초과 시 재시도 정책 적용, 반복 실패 항목은 별도 큐(dead-letter)에 보관 후 재처리
- 인증은 API Token + 서비스 어카운트 방식

### 7. 라벨링 습관화 방안

메타데이터 우선 판별로 누락에 대한 안전망이 생긴 만큼, 캠페인은 강제 차단보다
**넛지(선택을 쉽게 만들기) + 측정(채택률 추적) + 루틴화(지속 반복)** 조합으로 설계함.

- **넛지(UI)**: Jira 이슈 생성/전환 화면에서 `이슈 유형=Incident` 또는 `우선순위=Critical/High`인
  경우 `troubleshooting`/`incident` 라벨을 원클릭 추가 버튼으로 미리 노출 (직접 타이핑 요구 안 함)
- **메시지(WHY)**: "라벨 안 달면 누락"이 아니라 "라벨 달면 다음 사람이 검색으로 30분 절약한다"는
  본인 이득 중심 메시지로 캠페인 공지 작성
- **측정**: 메타데이터로 판별된 "장애 후보" 중 실제 라벨이 달린 비율을 **라벨링 채택률**로
  주간 추적, 부서별로 분리하여 반응 낮은 팀에 타겟 리마인드 발송
- **루틴화**: 신규 입사자 온보딩 체크리스트에 라벨링 규칙 포함, 분기별 채택률 리포트를
  전사 공지로 재발행 (1회성 캠페인이 아닌 반복 루틴)
- 워크플로우 차단(전환 자체를 막는 방식)은 메타데이터 안전망이 있는 한 우선순위를 낮춤 —
  반발이 크고, 자동 판별로 누락을 상당 부분 보완 가능하기 때문

## 제약조건

- MUST:
  - 자동 추출 결과 중 필수 5항목이 모두 채워지지 않으면 사람 검수 없이 등록 금지
  - 원문과 정규화 결과를 분리 저장
  - 동일 Jira 이슈/Confluence 페이지의 중복 등록 방지
- MUST NOT:
  - 자동 추출 라벨만으로 최종 라벨 확정 (사람 승인 없이)
  - 전체 히스토리를 단일 배치로 무제한 백필 (Rate Limit 위반 가능)
  - `staging` 풀의 애매한 판별 결과를 `confirmed`와 구분 없이 노출
- 완료조건:
  - Jira/Confluence 샘플 데이터로 polling → 메타데이터 판별 → (필요 시 LLM 보완) → staging/confirmed
    분리 저장까지의 흐름이 로그로 확인됨
  - 동일 이슈 재수집 시 신규 행이 아닌 갱신으로 처리됨을 확인
  - 라벨링 채택률 주간 리포트가 자동 생성됨을 확인

   데이터 현황 감사로 검증 — 담당: 파이프

4. 스케줄러·재처리 큐 설계 — 담당: 옵스
5. Jira 이슈 화면 라벨 원클릭 추천 UI 목업 — 담당: 픽셀
6. 라벨링 채택률 주간 리포트 설계 — 담당: 파이프
