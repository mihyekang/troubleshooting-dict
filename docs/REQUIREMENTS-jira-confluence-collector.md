# Jira/Confluence 이슈·리스크 수집 요구사항

## 배경

문제해결 사전은 현재 Open API로 개발자가 직접 이슈를 등록하는 구조만 갖춤.
하지만 사내 이슈·리스크 대부분은 Jira(장애 티켓)와 Confluence(포스트모템 문서)에 이미 누적되어 있어,
수동 재입력 대신 기존 데이터를 자동 수집하는 구조가 핵심 가치임.

## 참여자 의견 요약

| 팀원 | 핵심 의견 |
|------|----------|
| 스케치(AI 에이전트 아키텍트) | 자동 수집이 1순위, LLM 기반 5항목(현상/문제/영향도/해결방안/조치결과) 추출 정규화 필요 |
| 코어(백엔드 개발자) | Pull(배치 polling) 우선 구현, Webhook은 2단계, 중복 방지 key 필요 |
| 파이프(데이터 엔지니어) | 수집 전 데이터 현황 감사 선행, 원문/정규화 결과 분리 저장, 라벨은 1차 원본+2차 LLM 보완 |
| 옵스(DevOps 엔지니어) | 스케줄러 기반 증분 동기화, 실패 재처리(dead-letter), Rate Limit 대응 |

## 요구사항

### 1. 수집 방식

- **1단계(PoC)**: Jira REST API(JQL 검색), Confluence REST API(CQL 검색)를 주기적으로 polling하는 배치 수집기 구현
- **2단계**: Jira/Confluence Webhook으로 상태 변경 이벤트 즉시 수신 (보안팀과 인바운드 허용 협의 필요)
- 최초 1회 전체 백필(full backfill) 후, 이후에는 `updated` 시각 기준 증분 동기화

### 2. 수집 대상 필터

- Jira: 특정 라벨(`troubleshooting`, `incident`) 또는 특정 프로젝트의 `Resolved`/`Done` 상태 이슈
- Confluence: 특정 스페이스(예: `장애보고`, `포스트모템`) 하위 페이지

### 3. 데이터 정규화

- 수집한 원문(Jira 이슈 본문, Confluence 페이지 본문)은 가공 전 원문 그대로 별도 보관 (재처리 가능성 확보)
- LLM으로 원문에서 사전 필수 5항목 추출: 현상, 문제, 영향도, 해결방안, 조치결과
- 5항목 중 일부가 원문에 없으면 "추출 실패" 상태로 표시하고 사람 검수 큐로 전달 (자동 등록 금지)

### 4. 라벨링

- 1차: Jira 라벨/Confluence 페이지 라벨을 그대로 가져와 매핑
- 2차: LLM이 본문 기반 보완 라벨 제안 → 사람 승인 후 확정 (자동 라벨만으로 검색 품질 보장 불가)

### 5. 중복 방지 및 매핑

- `jira_issue_key`, `confluence_page_id`를 unique key로 사용
- 동일 key 재수집 시 신규 등록이 아닌 기존 이슈 갱신으로 처리

### 6. 운영 안정성

- 동기화 작업은 스케줄러(Cron 등)로 운영, 실패 시 알림 채널로 통보
- API 장애/Rate Limit 초과 시 재시도 정책 적용, 반복 실패 항목은 별도 큐(dead-letter)에 보관 후 재처리
- 인증은 API Token + 서비스 어카운트 방식

## 제약조건

- MUST:
  - 자동 추출 결과 중 필수 5항목이 모두 채워지지 않으면 사람 검수 없이 등록 금지
  - 원문과 정규화 결과를 분리 저장
  - 동일 Jira 이슈/Confluence 페이지의 중복 등록 방지
- MUST NOT:
  - 자동 추출 라벨만으로 최종 라벨 확정 (사람 승인 없이)
  - 전체 히스토리를 단일 배치로 무제한 백필 (Rate Limit 위반 가능)
- 완료조건:
  - Jira/Confluence 샘플 데이터로 polling → 정규화 → 검수 큐 등록까지의 흐름이 로그로 확인됨
  - 동일 이슈 재수집 시 신규 행이 아닌 갱신으로 처리됨을 확인
