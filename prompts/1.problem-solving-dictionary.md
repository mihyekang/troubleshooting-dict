[목표]
서비스 단위 이슈·리스크를 등록·조회·전사 공지할 수 있는 "문제해결 사전" 시스템의 설계·개발

[역할]
당신은 사내 운영 도구(이슈 트래커·온콜 시스템) 설계 경험 5년 이상을 보유한 백엔드/플랫폼 개발자입니다.
Open API 설계, 입력 스키마 표준화, 알림·공지 연동에 능숙합니다.

[맥락]
- 내 상황: 사내 여러 서비스에서 발생하는 이슈·리스크가 흩어져 기록되어 재발 시 과거 사례를 찾기 어려움.
  서비스별 발생 이슈를 한 곳에 축적하고, 개발자가 언제든 API로 상황을 등록하며,
  필요 시 전사 공지까지 이어지는 흐름을 만들고자 함
- 결과물 독자: 사내 개발자(이슈 등록자), 운영·SRE 담당자(조회·공지 발행자), 전사 직원(공지 수신자)

[입력정보]
- 해당 없음 (신규 구축, 작업 중 스키마·API 명세 직접 설계)

[작업방법]
1. 데이터 모델 설계
   - 이슈/리스크 엔티티 필드: `service`(서비스명), `severity`(CRITICAL/HIGH/MEDIUM/LOW),
     `title`, `description`, `cause`(원인), `resolution`(해결방법), `status`(OPEN/RESOLVED/MONITORING),
     `reporter`, `occurred_at`, `resolved_at`, `tags`(검색용 키워드 배열)
   - 동일 서비스·유사 키워드 이슈는 "관련 이슈"로 연결 가능하도록 참조 필드 포함
2. Open API 설계 (REST)
   - `POST /api/v1/issues` — 이슈 등록 (입력 포맷은 [작업방법] 1번 필드 기준 JSON)
   - `GET /api/v1/issues?service=&severity=&status=&keyword=` — 조회·검색
   - `PATCH /api/v1/issues/{id}` — 상태·해결방법 갱신
   - `POST /api/v1/issues/{id}/broadcast` — 등록된 이슈를 전사 공지로 발행 (승인자 권한 필요)
   - 인증: API Key 헤더 방식, 서비스별 키 발급
3. 입력 포맷 템플릿 제공
   - 개발자가 형식 고민 없이 채울 수 있는 JSON/웹폼 템플릿 제공 (필수: service, severity, title, description)
   - 누락 필드는 API 응답에서 400 에러와 누락 항목 목록으로 안내
4. 전사 공지 연동
   - `broadcast` 호출 시 사내 메신저/이메일 채널로 정해진 양식(서비스명·심각도·요약·해결방법) 발송
   - 공지 발행 이력은 별도 로그로 보관 (누가·언제·어떤 이슈를 공지했는지)
5. 검색·재발 방지 활용
   - 동일 `tags` 또는 `service`+`title` 유사도로 과거 유사 이슈 자동 추천 (등록 시점에 노출)
- 출력파일: `src/`
- 톤앤매너: API 명세는 표 형식, 코드 주석은 간결한 한국어
- 작성 규칙:
  - Python(FastAPI) 기반으로 작성
  - API Key는 `.env`에서 로드, 소스 하드코딩 금지
  - 추가 정보·의사결정 필요 시 작업 전 먼저 요청

[출력]
- `src/main.py` (FastAPI 앱)
- `src/schemas.py` (입력 포맷 정의)
- `docs/API.md` (Open API 명세)
- `README.md`

[제약조건]
- MUST:
  - 이슈 등록 API는 인증 없이 호출되지 않도록 API Key 검증 필수
  - 전사 공지 발행은 별도 권한 검증 후에만 허용
  - 입력 포맷 검증 실패 시 누락·오류 필드를 명시한 에러 응답 반환
- MUST NOT:
  - API Key를 소스 코드에 하드코딩
  - 검증되지 않은 이슈를 자동으로 전사 공지에 발행
- 완료조건:
  - `POST /api/v1/issues`로 이슈 등록 후 `GET /api/v1/issues`로 조회되는 응답 로그 확인
  - `broadcast` 호출 시 공지 채널로 발송된 메시지 로그 또는 발송 결과 응답 확인

[예시]
(입력) POST /api/v1/issues
```json
{
  "service": "결제-API",
  "severity": "HIGH",
  "title": "특정 카드사 결제 시 타임아웃 발생",
  "description": "오후 트래픽 집중 시간대 카드사 PG 응답 지연으로 결제 실패율 급증",
  "cause": "PG사 응답 지연, 재시도 로직 미적용",
  "resolution": "타임아웃 5초→10초 상향, 1회 재시도 로직 추가",
  "status": "RESOLVED",
  "reporter": "코어",
  "tags": ["결제", "타임아웃", "PG연동"]
}
```
(출력) 201 Created, `id` 발급 및 등록 확인 응답
